# 프론트엔드-백엔드 협업 방식 가이드

## 제1장. 문서 목적과 적용 범위

### 1.1절. 문서 작성 목적

#### 1.1.1항. 목적

본 문서는 프론트엔드와 백엔드가 실제 개발 과정에서 어떤 순서와 방식으로 협업하는지 설명하기 위한 가이드이다.

본 프로젝트 팀원들은 개발 경험이 많지 않으므로, 이 문서는 이론적 설명보다 실제 작업 순서, 문서화 방식, GitHub 사용 방식, API 협업 방식을 이해하는 데 초점을 둔다.

#### 1.1.2항. 이 문서가 답하는 질문

- 프론트엔드와 백엔드는 각각 무엇을 담당하는가.
- API는 왜 먼저 정의해야 하는가.
- 백엔드가 완성되지 않았을 때 프론트엔드는 어떻게 작업하는가.
- GitHub에서 브랜치와 Pull Request는 왜 사용하는가.
- 프론트엔드와 백엔드는 같은 레포지토리에서 작업하는 것이 좋은가.
- 기능 하나를 만들 때 어떤 순서로 협업해야 하는가.

### 1.2절. 본 프로젝트에서의 적용 범위

#### 1.2.1항. 프로젝트 상황

본 프로젝트는 LLM 채팅에서 메시지 기준 브랜치를 만들고, 브랜치 관계를 그래프로 시각화하는 웹 서비스이다.

MVP의 핵심은 완성도 높은 모든 기능을 한 번에 만드는 것이 아니라, 사용자가 다음 흐름을 실제로 경험할 수 있게 만드는 것이다.

1. 사용자가 채팅 메시지를 입력한다.
2. 시스템이 응답을 표시한다.
3. 사용자가 특정 메시지에서 브랜치를 생성한다.
4. 사용자가 브랜치별 대화를 전환한다.
5. 사용자가 그래프에서 브랜치 관계를 확인한다.

#### 1.2.2항. 적용 대상

이 문서는 다음 작업에 적용한다.

- 기능 요구사항 정리
- 화면 흐름 정리
- API 문서 작성
- 프론트엔드 구현
- 백엔드 구현
- GitHub 브랜치 관리
- Pull Request 리뷰
- 통합 테스트

### 1.3절. MVP 개발에서 우선할 기준

#### 1.3.1항. 기능적 요구사항 우선 원칙

MVP 단계에서는 기능적 요구사항을 우선 정리해야 한다.

기능적 요구사항은 사용자가 서비스에서 직접 수행할 수 있어야 하는 동작을 의미한다. 예를 들어 "사용자는 메시지를 입력할 수 있어야 한다", "사용자는 특정 메시지에서 브랜치를 생성할 수 있어야 한다", "사용자는 그래프에서 브랜치를 클릭해 이동할 수 있어야 한다"가 기능적 요구사항이다.

#### 1.3.2항. 최소 비기능 요구사항의 예외

MVP라고 해서 비기능 요구사항을 전부 무시해서는 안 된다. 서비스가 동작하지 않거나 테스트를 방해하는 수준의 요구사항은 최소한으로 정리해야 한다.

최소 비기능 요구사항의 예시는 다음과 같다.

- 사용자 데이터가 새로고침 후에도 유지되어야 하는가.
- API 오류가 발생했을 때 화면에 어떤 메시지를 보여줘야 하는가.
- 응답 속도가 너무 느릴 때 로딩 상태를 보여줘야 하는가.
- LLM API 비용이 과도하게 발생하지 않도록 제한이 필요한가.

따라서 본 프로젝트에서는 기능적 요구사항을 우선하되, 개발과 테스트를 깨뜨리는 최소 비기능 요구사항은 함께 기록하는 방식이 적절하다.

## 제2장. 프론트엔드와 백엔드의 역할

### 2.1절. 프론트엔드의 주요 역할

#### 2.1.1항. 화면 구현

프론트엔드는 사용자가 직접 보는 화면을 구현한다.

본 프로젝트에서 프론트엔드가 담당하는 화면의 예시는 다음과 같다.

- 채팅 화면
- 메시지 입력창
- 메시지 목록
- 브랜치 생성 버튼
- 현재 브랜치 표시
- 브랜치 그래프
- 브랜치 요약 미리보기
- 로딩 상태와 오류 상태

#### 2.1.2항. 사용자 행동 처리

프론트엔드는 사용자의 클릭, 입력, 선택, 이동을 처리한다.

예를 들어 사용자가 "브랜치 생성" 버튼을 클릭하면 프론트엔드는 다음 작업을 수행한다.

1. 어떤 메시지에서 브랜치를 생성하는지 확인한다.
2. 백엔드 API에 브랜치 생성 요청을 보낸다.
3. 응답으로 받은 새 브랜치 정보를 화면 상태에 반영한다.
4. 브랜치 그래프를 다시 그린다.

#### 2.1.3항. API 호출

프론트엔드는 백엔드가 제공하는 API를 호출한다. 프론트엔드는 데이터베이스에 직접 접근하지 않는다.

프론트엔드는 API 문서를 기준으로 요청을 보내고, 응답을 받아 화면에 표시한다.

### 2.2절. 백엔드의 주요 역할

#### 2.2.1항. 데이터 관리

백엔드는 서비스의 데이터를 저장하고 관리한다.

본 프로젝트에서 백엔드가 관리할 수 있는 데이터의 예시는 다음과 같다.

- 사용자 정보
- 채팅 세션
- 메시지
- 브랜치
- 브랜치의 부모 메시지
- 브랜치 활성 상태
- 브랜치 요약
- 이벤트 로그

#### 2.2.2항. API 제공

백엔드는 프론트엔드가 사용할 API를 제공한다.

예를 들어 프론트엔드가 메시지를 전송하면 백엔드는 다음 작업을 수행할 수 있다.

1. 사용자 메시지를 저장한다.
2. Mock LLM 또는 실제 LLM API를 호출한다.
3. AI 응답 메시지를 저장한다.
4. 저장된 메시지와 응답을 프론트엔드에 반환한다.

#### 2.2.3항. 비즈니스 로직 처리

백엔드는 서비스의 핵심 규칙을 처리한다.

본 프로젝트의 비즈니스 로직 예시는 다음과 같다.

- 브랜치는 반드시 특정 부모 메시지에서 생성되어야 한다.
- 루트 브랜치는 삭제하거나 비활성화하지 않는다.
- 비활성화된 브랜치는 삭제되지 않고 상태만 변경된다.
- 브랜치 그래프는 부모 브랜치와 자식 브랜치 관계를 기준으로 구성된다.

### 2.3절. 함께 합의해야 하는 영역

#### 2.3.1항. API 계약

API 계약은 프론트엔드와 백엔드가 함께 지켜야 하는 약속이다.

API 계약에는 다음 내용이 포함되어야 한다.

- API 주소
- HTTP 메서드
- 요청 데이터 형식
- 응답 데이터 형식
- 오류 응답 형식
- 인증 필요 여부
- 변경 이력

#### 2.3.2항. 데이터 이름과 의미

프론트엔드와 백엔드는 같은 데이터 이름을 같은 의미로 사용해야 한다.

예를 들어 `branchId`, `parentBranchId`, `parentMessageId`, `isActive`의 의미가 팀 내에서 다르게 해석되면 통합 단계에서 문제가 발생한다.

#### 2.3.3항. 완료 기준

프론트엔드와 백엔드는 기능이 언제 완료된 것으로 볼지 합의해야 한다.

단순히 코드 작성이 끝났다는 사실만으로 기능 완료라고 판단해서는 안 된다. API 문서 반영, 화면 연결, 오류 처리, 통합 테스트까지 끝난 상태를 완료 기준으로 삼아야 한다.

## 제3장. MVP 개발 순서

### 3.1절. 요구사항 정리

#### 3.1.1항. 요구사항 정리의 목적

요구사항 정리는 개발자가 무엇을 만들지 명확히 정하는 단계이다.

요구사항이 모호하면 프론트엔드는 다른 화면을 만들고, 백엔드는 다른 API를 만들게 된다. 따라서 개발 전에는 기능 단위로 요구사항을 짧고 검증 가능하게 정리해야 한다.

#### 3.1.2항. 좋은 요구사항의 기준

좋은 요구사항은 다음 조건을 만족해야 한다.

- 사용자의 행동이 포함되어 있다.
- 시스템의 반응이 포함되어 있다.
- 완료 여부를 확인할 수 있다.
- 한 문장에 여러 기능을 섞지 않는다.

예시는 다음과 같다.

```md
사용자는 특정 메시지에서 새 브랜치를 생성할 수 있어야 한다.
브랜치 생성 시 부모 브랜치 ID와 부모 메시지 ID가 저장되어야 한다.
```

### 3.2절. 화면 흐름 정리

#### 3.2.1항. 화면 흐름의 의미

화면 흐름은 사용자가 어떤 순서로 화면을 사용하게 되는지 정리한 것이다.

본 프로젝트의 기본 화면 흐름은 다음과 같다.

1. 사용자가 채팅 화면에 진입한다.
2. 사용자가 메시지를 입력한다.
3. AI 응답이 표시된다.
4. 사용자가 특정 메시지에서 브랜치를 생성한다.
5. 새 브랜치 화면으로 이동한다.
6. 그래프에서 브랜치 관계를 확인한다.

#### 3.2.2항. 화면 흐름 정리의 효과

화면 흐름을 먼저 정리하면 프론트엔드는 필요한 UI를 예상할 수 있고, 백엔드는 필요한 API를 예상할 수 있다.

### 3.3절. API 계약 정의

#### 3.3.1항. API를 먼저 정해야 하는 이유

프론트엔드와 백엔드는 API를 기준으로 병렬 작업을 진행한다.

API가 정해지지 않으면 프론트엔드는 어떤 데이터를 요청해야 하는지 알 수 없고, 백엔드는 어떤 형식으로 응답해야 하는지 알 수 없다.

#### 3.3.2항. API 계약 확정 시점

API 계약은 실제 백엔드 구현이 끝난 뒤가 아니라, 프론트엔드와 백엔드가 작업을 시작하기 전에 초안이 있어야 한다.

초안은 완벽할 필요가 없다. 그러나 최소한 다음 정보는 있어야 한다.

- 어떤 기능을 위한 API인지
- 어떤 주소로 호출하는지
- 어떤 값을 요청에 담는지
- 어떤 값을 응답으로 받는지

### 3.4절. 프론트엔드와 백엔드 병렬 개발

#### 3.4.1항. 병렬 개발의 의미

병렬 개발은 백엔드가 완성될 때까지 프론트엔드가 기다리지 않고, 프론트엔드와 백엔드가 동시에 개발하는 방식이다.

프론트엔드는 API 문서를 기준으로 Mock 데이터를 사용해 화면을 만든다. 백엔드는 같은 API 문서를 기준으로 실제 API를 구현한다.

#### 3.4.2항. 병렬 개발의 장점

병렬 개발을 하면 MVP 제작 속도가 빨라진다.

특히 본 프로젝트처럼 사용자 테스트를 위한 빠른 프로토타입이 중요한 경우, 프론트엔드는 Mock API로 먼저 사용 흐름을 검증하고 백엔드는 핵심 데이터 저장과 API 구현을 병행하는 방식이 적절하다.

### 3.5절. 통합 테스트와 수정

#### 3.5.1항. 통합 테스트의 의미

통합 테스트는 프론트엔드 화면과 백엔드 API를 실제로 연결해 기능이 끝까지 동작하는지 확인하는 단계이다.

#### 3.5.2항. 통합 테스트에서 확인할 항목

통합 테스트에서는 다음을 확인해야 한다.

- 프론트엔드 요청 형식이 API 문서와 일치하는가.
- 백엔드 응답 형식이 API 문서와 일치하는가.
- 성공 응답이 화면에 정상 표시되는가.
- 오류 응답이 화면에 정상 표시되는가.
- 새로고침 후에도 필요한 데이터가 유지되는가.

## 제4장. API 협업 방식

### 4.1절. API 문서의 의미

#### 4.1.1항. API 문서는 계약이다

API 문서는 프론트엔드와 백엔드 사이의 계약이다.

프론트엔드는 API 문서를 보고 화면에서 어떤 요청을 보낼지 구현한다. 백엔드는 API 문서를 보고 어떤 응답을 반환할지 구현한다.

#### 4.1.2항. API 문서가 없을 때 발생하는 문제

API 문서가 없으면 다음 문제가 발생한다.

- 프론트엔드가 임의로 요청 형식을 만든다.
- 백엔드가 임의로 응답 형식을 만든다.
- 통합 단계에서 데이터 이름이 맞지 않는다.
- 오류 처리 방식이 화면과 서버에서 다르게 구현된다.
- 기능은 각자 완성했지만 실제로는 연결되지 않는다.

### 4.2절. API 문서에 포함해야 할 항목

#### 4.2.1항. 기본 항목

API 문서에는 최소한 다음 항목이 있어야 한다.

| 항목 | 설명 |
| --- | --- |
| 기능 이름 | API가 어떤 기능을 위한 것인지 설명한다. |
| HTTP 메서드 | `GET`, `POST`, `PATCH`, `DELETE` 중 어떤 방식인지 적는다. |
| URL | API 주소를 적는다. |
| 요청 값 | 프론트엔드가 보내야 하는 값을 적는다. |
| 응답 값 | 백엔드가 반환해야 하는 값을 적는다. |
| 오류 응답 | 실패 시 반환되는 형식을 적는다. |
| 비고 | 인증, 제한사항, 후속 논의 사항을 적는다. |

#### 4.2.2항. 변경 이력

API 문서에는 변경 이력을 남겨야 한다.

예시는 다음과 같다.

```md
2026-06-27: 브랜치 생성 API 응답에 parentMessagePreview 필드를 추가했다.
```

### 4.3절. 요청과 응답 JSON 예시

#### 4.3.1항. 메시지 전송 API 예시

```md
기능: 채팅 메시지 전송
메서드: POST
URL: /api/sessions/{sessionId}/messages
```

요청 예시는 다음과 같다.

```json
{
  "branchId": "branch-root",
  "content": "브랜치 기반 채팅 UI의 장점을 설명해줘."
}
```

응답 예시는 다음과 같다.

```json
{
  "userMessage": {
    "id": "msg-101",
    "branchId": "branch-root",
    "role": "user",
    "content": "브랜치 기반 채팅 UI의 장점을 설명해줘.",
    "createdAt": "2026-06-27T10:00:00Z"
  },
  "assistantMessage": {
    "id": "msg-102",
    "branchId": "branch-root",
    "role": "assistant",
    "content": "브랜치 기반 채팅 UI는 꼬리질문을 별도 흐름으로 분리할 수 있다는 장점이 있다.",
    "createdAt": "2026-06-27T10:00:02Z"
  }
}
```

#### 4.3.2항. 브랜치 생성 API 예시

```md
기능: 특정 메시지에서 브랜치 생성
메서드: POST
URL: /api/branches
```

요청 예시는 다음과 같다.

```json
{
  "sessionId": "session-001",
  "parentBranchId": "branch-root",
  "parentMessageId": "msg-102",
  "title": "브랜치 UI 장점 추가 질문"
}
```

응답 예시는 다음과 같다.

```json
{
  "branch": {
    "id": "branch-201",
    "sessionId": "session-001",
    "parentBranchId": "branch-root",
    "parentMessageId": "msg-102",
    "title": "브랜치 UI 장점 추가 질문",
    "isActive": true,
    "isHidden": false,
    "summary": "",
    "createdAt": "2026-06-27T10:01:00Z"
  }
}
```

#### 4.3.3항. 오류 응답 예시

오류 응답 형식도 통일해야 한다.

```json
{
  "error": {
    "code": "PARENT_MESSAGE_NOT_FOUND",
    "message": "부모 메시지를 찾을 수 없다."
  }
}
```

### 4.4절. API 변경 시 지켜야 할 규칙

#### 4.4.1항. 문서 우선 규칙

API를 바꿔야 한다면 먼저 API 문서를 수정하거나, 코드 변경과 같은 Pull Request에서 API 문서를 함께 수정해야 한다.

문서 없이 API만 바꾸면 프론트엔드와 백엔드가 서로 다른 기준으로 작업하게 된다.

#### 4.4.2항. 공유 규칙

API 변경이 발생하면 팀 채팅 또는 회의에서 다음 내용을 공유해야 한다.

- 어떤 API가 바뀌었는가.
- 요청 값이 바뀌었는가.
- 응답 값이 바뀌었는가.
- 프론트엔드 수정이 필요한가.
- 기존 Mock 데이터도 수정해야 하는가.

## 제5장. GitHub 협업 방식

### 5.1절. 단일 레포와 분리 레포의 차이

#### 5.1.1항. 단일 레포 방식

단일 레포 방식은 프론트엔드, 백엔드, 문서를 하나의 GitHub 레포지토리에서 관리하는 방식이다.

예시는 다음과 같다.

```text
/
  frontend/
  backend/
  docs/
  README.md
  commit_log.md
```

장점은 다음과 같다.

- 프론트엔드와 백엔드 변경사항을 한 PR에서 함께 볼 수 있다.
- API 문서와 코드 변경을 함께 관리하기 쉽다.
- 초보 팀원이 레포지토리를 여러 개 관리하지 않아도 된다.
- 프로젝트 전체 흐름을 한 곳에서 파악하기 쉽다.

단점은 다음과 같다.

- 프로젝트가 커지면 폴더 구조 관리가 필요하다.
- 프론트엔드와 백엔드 배포 방식이 달라질 때 설정이 복잡해질 수 있다.

#### 5.1.2항. 분리 레포 방식

분리 레포 방식은 프론트엔드와 백엔드를 각각 다른 GitHub 레포지토리에서 관리하는 방식이다.

예시는 다음과 같다.

```text
llm-branch-chat-frontend
llm-branch-chat-backend
```

장점은 다음과 같다.

- 프론트엔드와 백엔드 배포를 독립적으로 관리하기 쉽다.
- 팀 규모가 커졌을 때 권한과 책임을 분리하기 쉽다.

단점은 다음과 같다.

- API 변경을 추적하기 어렵다.
- 문서가 어느 레포에 있어야 하는지 혼란이 생길 수 있다.
- 초보 팀에게는 GitHub 관리 부담이 커진다.

### 5.2절. 본 프로젝트에 적합한 레포 구조

#### 5.2.1항. 권장 방식

본 프로젝트는 단일 레포 방식을 권장한다.

그 이유는 다음과 같다.

- 팀원들이 개발 경험이 많지 않다.
- MVP를 빠르게 만들어야 한다.
- 프론트엔드와 백엔드가 API 문서를 중심으로 자주 맞춰야 한다.
- 기획 문서, API 문서, 코드 변경을 한 곳에서 추적하는 편이 안전하다.

#### 5.2.2항. 권장 폴더 구조

향후 프론트엔드와 백엔드를 명확히 분리할 경우 다음 구조를 사용할 수 있다.

```text
/
  frontend/
    src/
    tests/
    package.json
  backend/
    src/
    tests/
    package.json
  docs/
    PRD.md
    requirements.md
    api.md
    implementation_plan.md
    test_plan.md
  commit_log.md
  README.md
```

현재 저장소 구조와 다를 수 있으므로, 이 구조는 향후 구조화 예시로 본다.

### 5.3절. 브랜치 사용 규칙

#### 5.3.1항. main 브랜치 보호 원칙

`main` 브랜치는 항상 실행 가능한 상태로 유지해야 한다.

개발자는 `main` 브랜치에 직접 코드를 작성하지 않는다. 기능 개발, 오류 수정, 문서 수정은 별도 브랜치에서 진행한다.

#### 5.3.2항. 브랜치 이름 규칙

브랜치 이름은 작업 목적이 드러나게 작성한다.

권장 규칙은 다음과 같다.

| 작업 유형 | 브랜치 예시 |
| --- | --- |
| 기능 개발 | `feature/create-branch-api` |
| 화면 개발 | `feature/branch-graph-ui` |
| 오류 수정 | `fix/message-submit-error` |
| 문서 수정 | `docs/api-contract` |
| 실험 작업 | `experiment/graph-layout` |

#### 5.3.3항. 실험 코드 관리 방식

개발자가 로컬에서 실험하고 싶은 코드는 별도 브랜치에서 작업한다.

실험 브랜치는 다음 기준으로 관리한다.

- 개인 실험은 로컬 브랜치에서 진행한다.
- 팀원에게 보여줄 필요가 있으면 원격 GitHub에 push한다.
- 실험 결과가 실제 기능으로 채택되면 새 기능 브랜치로 정리하거나 PR을 만든다.
- 실패한 실험 브랜치는 main에 합치지 않는다.

### 5.4절. Pull Request 사용 규칙

#### 5.4.1항. Pull Request의 의미

Pull Request는 내가 만든 변경사항을 `main` 브랜치에 합치기 전에 팀원에게 검토받는 절차이다.

PR은 단순한 형식이 아니라 다음을 확인하는 협업 장치이다.

- 기능이 요구사항과 맞는가.
- API 문서와 코드가 일치하는가.
- 다른 기능을 깨뜨리지 않는가.
- 팀원이 이해할 수 있는 코드인가.

#### 5.4.2항. PR에 포함해야 할 내용

PR 본문에는 다음 내용을 포함해야 한다.

```md
## 작업 내용
- 브랜치 생성 API를 추가했다.
- 부모 메시지 ID와 부모 브랜치 ID를 저장하도록 구현했다.

## 확인 방법
- 브랜치 생성 요청을 보내면 새 branchId가 반환되는지 확인한다.
- 존재하지 않는 parentMessageId 요청 시 오류 응답이 반환되는지 확인한다.

## 문서 변경
- docs/api.md에 브랜치 생성 API 요청/응답 예시를 추가했다.

## 관련 요구사항
- REQ-004
- REQ-005
```

#### 5.4.3항. PR 크기 기준

PR은 가능한 작게 만든다.

좋은 PR 예시는 다음과 같다.

- 브랜치 생성 API만 구현한 PR
- 브랜치 그래프 UI만 구현한 PR
- API 문서만 정리한 PR

나쁜 PR 예시는 다음과 같다.

- 로그인, 채팅, 브랜치, 그래프, 디자인 수정을 한 번에 포함한 PR
- 기능 변경과 대규모 코드 정리를 섞은 PR
- API 변경을 했지만 문서 변경이 없는 PR

### 5.5절. 코드 리뷰 기준

#### 5.5.1항. 리뷰어가 확인할 항목

코드 리뷰에서는 다음을 확인한다.

- 요구사항과 맞게 구현되었는가.
- API 문서와 코드가 일치하는가.
- 오류 상황이 처리되었는가.
- 불필요하게 큰 변경이 섞이지 않았는가.
- 변수명과 함수명이 이해 가능한가.
- 테스트 또는 수동 확인 방법이 적혀 있는가.

#### 5.5.2항. 리뷰 코멘트 작성 방식

리뷰 코멘트는 문제와 이유를 함께 적는다.

예시는 다음과 같다.

```md
parentMessageId가 없는 경우에도 브랜치가 생성될 수 있는 상태이다.
요구사항상 브랜치는 특정 메시지에서 생성되어야 하므로, 백엔드에서 parentMessageId 검증이 필요하다.
```

## 제6장. 프론트엔드와 백엔드의 실제 작업 흐름

### 6.1절. 백엔드 API 완성 전 프론트엔드 작업 방식

#### 6.1.1항. Mock 데이터 사용

백엔드 API가 아직 완성되지 않아도 프론트엔드는 Mock 데이터를 사용해 화면을 먼저 만들 수 있다.

Mock 데이터는 실제 API 응답과 같은 형태여야 한다.

예시는 다음과 같다.

```json
{
  "id": "branch-201",
  "title": "브랜치 UI 장점 추가 질문",
  "parentBranchId": "branch-root",
  "parentMessageId": "msg-102",
  "isActive": true
}
```

#### 6.1.2항. Mock 데이터 사용 시 주의점

Mock 데이터는 화면만 만들기 위한 임시 데이터이다.

다만 실제 API와 다른 형태의 Mock 데이터를 사용하면 통합 단계에서 문제가 생긴다. 따라서 Mock 데이터는 반드시 API 문서와 같은 필드명을 사용해야 한다.

### 6.2절. 백엔드 구현 방식

#### 6.2.1항. API 문서 기준 구현

백엔드는 API 문서에 적힌 요청과 응답 형식을 기준으로 구현한다.

구현 중 API 문서와 맞지 않는 부분이 발견되면, 백엔드가 임의로 바꾸지 않고 프론트엔드 담당자와 논의한 뒤 문서를 수정해야 한다.

#### 6.2.2항. 데이터 검증

백엔드는 잘못된 요청을 검증해야 한다.

예시는 다음과 같다.

- 존재하지 않는 세션 ID로 요청한 경우 오류를 반환한다.
- 존재하지 않는 부모 메시지 ID로 브랜치를 만들 수 없게 한다.
- 이미 비활성화된 브랜치를 다시 비활성화할 때 처리 방식을 정한다.

### 6.3절. 백엔드 구현 완료 후 연결 방식

#### 6.3.1항. Mock API 제거 또는 전환

백엔드 API가 완성되면 프론트엔드는 Mock 데이터를 실제 API 호출로 교체한다.

이때 화면 구조를 크게 바꾸지 않고 데이터 호출 부분만 바꾸는 것이 좋은 구조이다.

#### 6.3.2항. 연결 후 확인할 항목

연결 후에는 다음을 확인한다.

- 메시지 전송이 실제 API로 처리되는가.
- 브랜치 생성 후 새 브랜치가 화면에 표시되는가.
- 브랜치 그래프가 백엔드 데이터 기준으로 갱신되는가.
- API 오류가 발생했을 때 사용자에게 적절한 안내가 표시되는가.

### 6.4절. 통합 과정에서 자주 생기는 문제

#### 6.4.1항. 필드명 불일치

가장 자주 발생하는 문제는 필드명 불일치이다.

예를 들어 프론트엔드는 `parentMessageId`를 기대하는데 백엔드가 `parent_message_id`를 반환하면 화면에서 데이터를 읽지 못할 수 있다.

#### 6.4.2항. 응답 구조 불일치

프론트엔드가 배열을 기대하는데 백엔드가 객체를 반환하거나, 백엔드가 중첩 객체를 반환하는데 프론트엔드가 평면 구조를 기대하면 문제가 생긴다.

#### 6.4.3항. 오류 처리 누락

성공 상황만 구현하고 오류 상황을 처리하지 않으면 사용자 테스트 중 문제가 발생한다.

최소한 다음 오류는 처리해야 한다.

- 네트워크 오류
- 서버 오류
- 요청 값 누락
- 존재하지 않는 데이터 요청
- LLM API 응답 실패

## 제7장. 작업 완료 기준

### 7.1절. 기능 개발 완료 기준

#### 7.1.1항. 완료의 정의

기능 개발 완료는 코드 작성 완료가 아니다.

다음 조건을 만족해야 완료로 판단한다.

- 요구사항에 맞게 동작한다.
- 화면에서 사용자가 실제로 사용할 수 있다.
- 백엔드 API와 연결되어 있다.
- 오류 상황이 최소한으로 처리되어 있다.
- API 문서가 최신 상태이다.
- PR 리뷰를 통과했다.

### 7.2절. API 변경 완료 기준

#### 7.2.1항. API 변경 완료 조건

API 변경은 다음 조건을 만족해야 완료로 본다.

- API 문서가 수정되었다.
- 프론트엔드 Mock 데이터가 수정되었다.
- 백엔드 응답 형식이 수정되었다.
- 관련 화면이 정상 동작한다.
- 팀원에게 변경 사실이 공유되었다.

### 7.3절. PR 완료 기준

#### 7.3.1항. Merge 전 확인 항목

PR을 merge하기 전에는 다음을 확인한다.

- PR 본문에 작업 내용이 적혀 있다.
- 확인 방법이 적혀 있다.
- 관련 문서 변경이 포함되어 있다.
- 최소 1명 이상의 팀원이 리뷰했다.
- 충돌이 없다.
- 실행 또는 테스트 결과가 공유되었다.

### 7.4절. 팀 공유 완료 기준

#### 7.4.1항. 공유해야 하는 내용

작업이 끝나면 다음 내용을 팀에 공유한다.

- 어떤 기능이 완료되었는가.
- 어떤 API가 추가되거나 변경되었는가.
- 프론트엔드 또는 백엔드에서 후속 작업이 필요한가.
- 테스트 방법은 무엇인가.
- 아직 남은 문제는 무엇인가.

## 제8장. 팀 커뮤니케이션 규칙

### 8.1절. 데일리 체크 방식

#### 8.1.1항. 공유 항목

짧은 데일리 체크에서는 다음만 공유해도 충분하다.

- 어제 완료한 작업
- 오늘 진행할 작업
- 막힌 부분
- 다른 팀원에게 필요한 도움

### 8.2절. API 변경 공유 방식

#### 8.2.1항. API 변경 메시지 형식

API가 변경되면 다음 형식으로 공유한다.

```md
[API 변경]
- 대상 API: POST /api/branches
- 변경 내용: 응답에 parentMessagePreview 필드 추가
- 프론트엔드 영향: 브랜치 상세 표시에서 사용 가능
- 문서 반영: docs/api.md 수정 완료
- 적용 PR: #12
```

### 8.3절. 막힌 문제 공유 방식

#### 8.3.1항. 문제 공유 시 포함할 내용

막힌 문제를 공유할 때는 다음을 포함한다.

- 어떤 작업 중이었는가.
- 어떤 결과를 기대했는가.
- 실제로 어떤 문제가 발생했는가.
- 어떤 파일 또는 API와 관련이 있는가.
- 이미 시도한 해결 방법은 무엇인가.

### 8.4절. 회의에서 결정해야 할 사항

#### 8.4.1항. 회의 안건 기준

회의에서는 취향보다 결정이 필요한 사안을 다룬다.

예시는 다음과 같다.

- 브랜치 생성 버튼을 메시지마다 둘 것인가.
- 브랜치 그래프를 항상 보여줄 것인가, 팝오버로 보여줄 것인가.
- 비활성 브랜치를 숨겼을 때 복구 버튼은 어디에 둘 것인가.
- 실제 LLM API를 MVP에서 연결할 것인가, Mock으로 사용자 테스트를 먼저 할 것인가.

## 제9장. 본 프로젝트 적용 예시

### 9.1절. 브랜치 생성 기능 개발 예시

#### 9.1.1항. 기능 요구사항

브랜치 생성 기능의 요구사항은 다음과 같이 정리할 수 있다.

```md
사용자는 특정 메시지에서 새 브랜치를 생성할 수 있어야 한다.
새 브랜치는 부모 브랜치 ID와 부모 메시지 ID를 저장해야 한다.
브랜치 생성 후 사용자는 새 브랜치 대화 화면으로 이동해야 한다.
그래프에는 새 브랜치 노드와 부모 브랜치에서 이어지는 엣지가 표시되어야 한다.
```

### 9.2절. 프론트엔드 작업 항목

#### 9.2.1항. 작업 목록

프론트엔드는 다음을 구현한다.

- 메시지 옆에 브랜치 생성 버튼을 표시한다.
- 버튼 클릭 시 현재 메시지 ID와 현재 브랜치 ID를 가져온다.
- 브랜치 생성 API를 호출한다.
- API 응답으로 받은 새 브랜치를 상태에 추가한다.
- 현재 브랜치를 새 브랜치로 전환한다.
- 그래프 UI를 갱신한다.
- 오류 발생 시 사용자에게 안내 메시지를 표시한다.

### 9.3절. 백엔드 작업 항목

#### 9.3.1항. 작업 목록

백엔드는 다음을 구현한다.

- 브랜치 생성 API를 만든다.
- 요청의 `sessionId`, `parentBranchId`, `parentMessageId`를 검증한다.
- 새 브랜치 ID를 생성한다.
- 부모 브랜치와 부모 메시지 관계를 저장한다.
- 생성된 브랜치 정보를 응답으로 반환한다.
- 잘못된 요청에 대한 오류 응답을 반환한다.

### 9.4절. API 문서 예시

#### 9.4.1항. 브랜치 생성 API 문서

```md
## 브랜치 생성

- 메서드: POST
- URL: /api/branches
- 설명: 특정 메시지를 기준으로 새 브랜치를 생성한다.

### 요청

{
  "sessionId": "session-001",
  "parentBranchId": "branch-root",
  "parentMessageId": "msg-102",
  "title": "추가 질문"
}

### 응답

{
  "branch": {
    "id": "branch-201",
    "sessionId": "session-001",
    "parentBranchId": "branch-root",
    "parentMessageId": "msg-102",
    "title": "추가 질문",
    "isActive": true,
    "isHidden": false,
    "summary": "",
    "createdAt": "2026-06-27T10:01:00Z"
  }
}

### 오류

{
  "error": {
    "code": "PARENT_MESSAGE_NOT_FOUND",
    "message": "부모 메시지를 찾을 수 없다."
  }
}
```

### 9.5절. PR 작성 예시

#### 9.5.1항. 프론트엔드 PR 예시

```md
## 작업 내용
- 메시지별 브랜치 생성 버튼을 추가했다.
- 브랜치 생성 API 호출 로직을 추가했다.
- 브랜치 생성 성공 시 현재 브랜치를 새 브랜치로 전환하도록 구현했다.

## 확인 방법
- 메시지 옆 브랜치 버튼을 클릭한다.
- 새 브랜치가 생성되고 그래프에 노드가 추가되는지 확인한다.

## 문서 변경
- API 변경 없음.
```

#### 9.5.2항. 백엔드 PR 예시

```md
## 작업 내용
- POST /api/branches API를 추가했다.
- 부모 브랜치 ID와 부모 메시지 ID 검증을 추가했다.
- 브랜치 생성 후 새 브랜치 정보를 반환하도록 구현했다.

## 확인 방법
- 정상 요청 시 branch 객체가 반환되는지 확인한다.
- 존재하지 않는 parentMessageId 요청 시 오류가 반환되는지 확인한다.

## 문서 변경
- docs/api.md에 브랜치 생성 API를 추가했다.
```

## 제10장. 초보 팀을 위한 핵심 정리

### 10.1절. 반드시 지켜야 할 원칙

#### 10.1.1항. 핵심 원칙

팀은 다음 원칙을 지켜야 한다.

- 요구사항을 먼저 정리하고 개발을 시작한다.
- API 문서를 프론트엔드와 백엔드의 계약으로 본다.
- API가 변경되면 문서도 함께 변경한다.
- `main` 브랜치에 직접 작업하지 않는다.
- 기능별 브랜치를 만들고 PR로 합친다.
- PR에는 작업 내용과 확인 방법을 적는다.
- 백엔드가 완성되기 전 프론트엔드는 Mock 데이터로 작업할 수 있다.
- 통합 테스트 전까지 기능이 끝났다고 판단하지 않는다.

### 10.2절. 실수하기 쉬운 지점

#### 10.2.1항. 자주 발생하는 실수

초보 팀에서 자주 발생하는 실수는 다음과 같다.

- 요구사항 없이 바로 코드를 작성한다.
- API 문서 없이 프론트엔드와 백엔드를 따로 만든다.
- Mock 데이터와 실제 API 응답 형식이 다르다.
- 각자 `main` 브랜치에 직접 push한다.
- PR이 너무 커서 리뷰가 어렵다.
- 오류 상황을 처리하지 않는다.
- API 변경 사실을 팀원에게 공유하지 않는다.

### 10.3절. 작업 시작 전 체크리스트

#### 10.3.1항. 공통 체크리스트

작업 시작 전 다음을 확인한다.

- 이 기능의 요구사항이 문서에 있는가.
- 이 기능에 필요한 화면 흐름이 정리되어 있는가.
- 이 기능에 필요한 API가 정의되어 있는가.
- 내가 작업할 브랜치를 만들었는가.
- 이 작업이 프론트엔드 작업인지, 백엔드 작업인지, 공통 작업인지 분류했는가.

#### 10.3.2항. 프론트엔드 체크리스트

프론트엔드 작업자는 다음을 확인한다.

- API 응답 예시를 확보했는가.
- 백엔드가 없을 때 사용할 Mock 데이터가 있는가.
- 로딩 상태와 오류 상태를 표시하는가.
- 화면에서 사용자가 다음 행동을 이해할 수 있는가.

#### 10.3.3항. 백엔드 체크리스트

백엔드 작업자는 다음을 확인한다.

- API 요청 값과 응답 값이 문서와 일치하는가.
- 잘못된 요청을 검증하는가.
- 프론트엔드가 필요한 데이터를 모두 반환하는가.
- API 변경이 문서에 반영되었는가.

## 제11장. 권장 운영 방식

### 11.1절. 본 프로젝트의 기본 운영안

#### 11.1.1항. 운영안

본 프로젝트는 다음 방식으로 운영하는 것이 적절하다.

- 레포지토리는 단일 레포 방식을 사용한다.
- 프론트엔드와 백엔드는 폴더를 나누어 관리한다.
- API 문서는 `docs/api.md` 또는 이에 준하는 문서에 관리한다.
- 기능 개발은 GitHub Flow 방식으로 진행한다.
- 모든 기능은 브랜치에서 작업하고 PR로 합친다.
- PR에는 문서 변경 여부를 반드시 적는다.
- API 변경은 코드 변경과 같은 PR에서 처리하거나, 문서 PR을 먼저 만든다.

### 11.2절. 추천 작업 흐름

#### 11.2.1항. 기능 하나를 개발할 때의 흐름

기능 하나를 개발할 때는 다음 순서를 따른다.

1. 요구사항을 한 문장으로 정리한다.
2. 화면에서 어떤 행동이 발생하는지 정리한다.
3. 필요한 API를 문서로 정의한다.
4. 프론트엔드는 Mock 데이터로 화면을 만든다.
5. 백엔드는 API 문서 기준으로 실제 API를 만든다.
6. 프론트엔드는 Mock 데이터를 실제 API 호출로 교체한다.
7. 통합 테스트를 진행한다.
8. PR을 만들고 리뷰를 받는다.
9. 문서와 코드가 함께 최신 상태인지 확인한다.
10. `main` 브랜치에 merge한다.

#### 11.2.2항. 최종 정리

프론트엔드와 백엔드 협업의 핵심은 "같은 문서를 보고, 작은 단위로 나누어, PR로 검증하면서 합치는 것"이다.

초보 팀일수록 구두 합의에 의존하면 문제가 커진다. 요구사항, API, PR, 변경 이력을 문서로 남기는 방식이 프로젝트 안정성을 높인다.
