# 요구사항 목록

## 개요

본 문서는 `docs/PRD.md`를 기준으로 MVP 요구사항을 ID 단위로 정리한 문서이다. 각 요구사항은 구현 계획, 테스트 계획, 충족 여부 추적의 기준으로 사용한다.

## 기능 요구사항

| ID | 요구사항 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| REQ-001 | 기본 LLM 채팅 인터페이스 | 사용자는 ChatGPT, Gemini, Claude와 유사한 대화형 채팅 UI에서 메시지를 주고받을 수 있어야 한다. | Must |
| REQ-002 | 메시지 단위 브랜치 생성 | 사용자는 특정 메시지를 기준으로 새 브랜치를 생성할 수 있어야 한다. | Must |
| REQ-003 | 브랜치 분기 지점 기록 | 시스템은 브랜치가 어느 메시지에서 분기되었는지 저장하고 표시할 수 있어야 한다. | Must |
| REQ-004 | 그래프 기반 브랜치 시각화 | 사용자는 전체 대화와 브랜치 관계를 그래프 형태로 확인할 수 있어야 한다. | Must |
| REQ-005 | 브랜치 클릭 이동 | 사용자는 그래프에서 브랜치를 클릭하여 해당 브랜치 대화로 이동할 수 있어야 한다. | Must |
| REQ-006 | 브랜치 비활성화 | 사용자는 필요 없는 브랜치를 삭제하지 않고 비활성 상태로 전환할 수 있어야 한다. | Must |
| REQ-007 | 비활성 브랜치 시각 표현 | 비활성 브랜치는 회색 표시, 접기, 숨김 등 낮은 강조 상태로 표현되어야 한다. | Must |
| REQ-008 | 비활성 브랜치 복구 가능성 | 접기 또는 숨김 처리된 브랜치는 사용자가 다시 확인하거나 활성화할 수 있어야 한다. | Should |
| REQ-009 | 그래프 확대 및 축소 | 사용자는 브랜치가 많아질 때 그래프를 확대하거나 축소할 수 있어야 한다. | Should |
| REQ-010 | 브랜치 요약 미리보기 | 사용자가 브랜치에 마우스를 올리면 LLM 자동 요약 기반 미리보기가 표시되어야 한다. | Should |
| REQ-011 | LLM API 연동 고려 구조 | MVP는 Mock 응답으로 시작할 수 있으나 실제 LLM API 연동이 가능한 구조로 설계되어야 한다. | Must |
| REQ-012 | 사용자 테스트 가능 품질 | 실제 사용자를 모집해 테스트할 수 있도록 기본 사용 흐름과 오류 처리가 안정적이어야 한다. | Should |

## 비기능 요구사항

| ID | 요구사항 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| NFR-001 | 대화 관계 데이터 안정성 | 메시지와 브랜치의 부모-자식 관계가 손실 없이 저장되어야 한다. | Must |
| NFR-002 | 그래프 가독성 | 브랜치 수가 늘어나도 현재 위치와 원래 목적을 파악할 수 있어야 한다. | Must |
| NFR-003 | 운영 비용 고려 | 실제 LLM API 사용 시 비용 관리가 가능하도록 호출 지점을 분리해야 한다. | Should |
| NFR-004 | 사용자 검증 데이터 확보 | 만족도 조사와 사용성 테스트 결과를 수집할 수 있는 운영 방안을 고려해야 한다. | Should |

| OOS-005 | 브랜치 병합 기능 | MVP의 핵심 문제는 분기 관리와 시각화이다. |
