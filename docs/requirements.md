# 구현 가능 요구사항 목록

## 개요

본 문서는 `docs/PRD.md`를 기준으로 MVP에서 구현 가능한 요구사항을 정리한 문서이다. 각 요구사항은 `REQ-001` 형식의 ID를 사용하며, 구현 완료 여부를 판단할 수 있도록 검증 기준을 함께 정의한다.

## 작성 기준

- PRD의 핵심 기능, MVP 범위, 개발 고려사항, 성공 기준에서 실제 구현 가능한 항목만 요구사항으로 분리했다.
- PRD의 범위 제외 항목은 구현 요구사항에서 제외했다.
- 그래프 UI의 세부 레이아웃처럼 아직 확정되지 않은 항목은 특정 디자인을 강제하지 않고 검증 가능한 동작 기준으로 작성했다.

## 요구사항 목록

| ID | 요구사항 | 구현 범위 | 검증 기준 | 우선순위 |
| --- | --- | --- | --- | --- |
| REQ-001 | 기본 채팅 세션을 제공해야 한다. | 사용자는 독립 웹 서비스의 채팅 화면에서 새 대화 세션을 시작할 수 있어야 한다. | 새 세션 진입 시 빈 채팅 화면이 표시되고, 사용자가 첫 메시지를 입력할 수 있으면 충족이다. | Must |
| REQ-002 | 사용자는 메시지를 입력하고 응답을 확인할 수 있어야 한다. | 사용자 메시지 입력, 전송, 응답 표시, 메시지 순서 유지를 구현해야 한다. | 메시지를 전송하면 사용자 메시지와 시스템 응답이 시간순으로 화면에 표시되고, 새로고침 또는 상태 갱신 후에도 순서가 유지되면 충족이다. | Must |
| REQ-003 | LLM 응답 제공 계층은 교체 가능해야 한다. | MVP에서는 Mock LLM을 사용할 수 있으나 실제 LLM API로 교체 가능한 인터페이스를 분리해야 한다. | 채팅 UI와 브랜치 로직을 수정하지 않고 Mock 응답 제공자를 실제 API 제공자로 교체할 수 있으면 충족이다. | Must |
| REQ-004 | 사용자는 특정 메시지에서 브랜치를 생성할 수 있어야 한다. | 각 메시지 또는 선택 가능한 메시지 영역에서 브랜치 생성 액션을 제공해야 한다. | 특정 메시지에서 브랜치 생성을 실행했을 때 고유 ID를 가진 새 브랜치가 생성되면 충족이다. | Must |
| REQ-005 | 브랜치의 분기 관계를 저장해야 한다. | 브랜치는 부모 브랜치 ID, 부모 메시지 ID, 생성 시각, 활성 상태를 저장해야 한다. | 생성된 브랜치 데이터에서 부모 브랜치와 부모 메시지를 조회할 수 있고, 해당 값이 실제 분기 지점과 일치하면 충족이다. | Must |
| REQ-006 | 사용자는 브랜치별 대화 흐름을 전환할 수 있어야 한다. | 현재 브랜치 상태를 관리하고, 선택한 브랜치의 메시지 흐름을 채팅 화면에 표시해야 한다. | 브랜치를 전환했을 때 채팅 화면이 해당 브랜치의 메시지 흐름으로 바뀌고 현재 브랜치 상태가 갱신되면 충족이다. | Must |
| REQ-007 | 브랜치 분기 지점을 화면에 표시해야 한다. | 사용자는 현재 브랜치가 어느 메시지에서 분기되었는지 확인할 수 있어야 한다. | 브랜치 상세 정보 또는 그래프에서 부모 메시지의 식별 정보나 요약을 확인할 수 있으면 충족이다. | Must |
| REQ-008 | 브랜치 관계를 그래프로 시각화해야 한다. | 브랜치 목록을 노드와 엣지로 변환하여 그래프 UI에 표시해야 한다. | 루트 브랜치 1개와 하위 브랜치 3개를 가진 테스트 데이터에서 노드 4개와 엣지 3개가 표시되면 충족이다. | Must |
| REQ-009 | 그래프에서 현재 브랜치를 식별할 수 있어야 한다. | 현재 선택된 브랜치를 다른 브랜치와 구분되는 시각 상태로 표시해야 한다. | 브랜치 전환 시 그래프의 현재 브랜치 강조 표시가 함께 변경되면 충족이다. | Must |
| REQ-010 | 사용자는 그래프에서 브랜치를 클릭해 이동할 수 있어야 한다. | 그래프 노드 클릭을 채팅 브랜치 전환 동작과 연결해야 한다. | 그래프 노드를 클릭했을 때 해당 브랜치의 대화 화면으로 이동하면 충족이다. | Must |
| REQ-011 | 사용자는 그래프를 확대하거나 축소할 수 있어야 한다. | 그래프 UI에 확대, 축소, 기본 배율 복귀 조작을 제공해야 한다. | 확대와 축소 조작 후 그래프 배율이 변경되고, 기본 배율 복귀 조작으로 초기 배율로 돌아오면 충족이다. | Should |
| REQ-012 | 사용자는 브랜치를 비활성화할 수 있어야 한다. | 브랜치를 삭제하지 않고 비활성 상태로 전환하는 기능을 제공해야 한다. | 비활성화 후에도 브랜치 데이터와 메시지가 유지되고, 활성 상태 값만 변경되면 충족이다. | Must |
| REQ-013 | 비활성 브랜치는 낮은 강조 상태로 표시되어야 한다. | 비활성 브랜치는 그래프 또는 목록에서 회색 표시 등 낮은 강조 상태로 표현되어야 한다. | 비활성화된 브랜치가 활성 브랜치와 시각적으로 구분되고, 다시 활성화하면 원래 강조 상태로 복구되면 충족이다. | Must |
| REQ-014 | 사용자는 비활성 브랜치를 접거나 숨길 수 있어야 한다. | 비활성 브랜치를 삭제하지 않고 그래프에서 접기 또는 숨김 처리할 수 있어야 한다. | 숨김 처리 후 브랜치가 그래프의 주요 표시 영역에서 제거되고, 사용자가 다시 표시하거나 활성화할 수 있으면 충족이다. | Should |
| REQ-015 | 브랜치 요약 미리보기를 제공해야 한다. | 브랜치별 요약을 저장하고, 사용자가 그래프에서 브랜치에 마우스를 올리면 요약을 표시해야 한다. | 브랜치 노드에 마우스를 올렸을 때 해당 브랜치의 요약 팝오버가 표시되고, 마우스를 벗어나면 사라지면 충족이다. | Should |
| REQ-016 | 사용자 테스트 지표를 수집할 수 있어야 한다. | 브랜치 생성, 브랜치 이동, 비활성화, 세션 이용 시간 등 성공 기준 검증에 필요한 이벤트를 기록할 수 있어야 한다. | 테스트 세션에서 주요 이벤트가 세션 ID, 이벤트명, 발생 시각과 함께 기록되면 충족이다. | Should |

## 범위 제외 항목

| 항목 | 제외 사유 |
| --- | --- |
| 복수 LLM 제공자 전체 연동 | MVP에서는 Mock LLM 또는 하나의 실제 LLM 제공자 연동 가능 구조까지만 요구한다. |
| ChatGPT 확장 프로그램 | 초기 제품 형태는 독립 웹 서비스이다. |
| 실제 서비스 수준의 LLM API 운영 안정화 | MVP에서는 API 교체 가능한 구조와 기본 오류 처리까지만 요구한다. |
| 협업 기능 | 개인 사용자의 대화 흐름 관리 검증이 우선이다. |
| 브랜치 자동 추천 | 사용자가 직접 브랜치를 생성하는 흐름부터 검증한다. |
| 모바일 최적화 | MVP의 우선 검증 대상은 데스크톱 기반 그래프 UI이다. |
| 고급 검색 및 필터링 | MVP의 핵심은 브랜치 생성과 시각화이다. |
| 브랜치 병합 기능 | MVP 범위에 포함되지 않는다. |
