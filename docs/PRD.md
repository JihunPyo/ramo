# PRD: LLM 채팅 브랜치 시각화 서비스

## 1. 프로젝트 개요

LLM과의 대화형 채팅에서 특정 메시지를 기준으로 브랜치를 생성하고, 브랜치 간 관계를 그래프 형태로 시각화하는 독립 웹 서비스이다. 사용자는 본래 대화 목적을 유지하면서도 새로 생긴 궁금증을 별도 브랜치에서 탐색할 수 있다.

장기적으로는 기존 ChatGPT에 확장 프로그램을 붙이는 방식도 가능하지만, 초기 프로젝트는 학회 프로젝트의 목적에 맞게 완성도 있는 독립 웹 서비스를 만드는 방향으로 진행한다. 목표는 데모 수준이 아니라 실제 사용자를 모집하고 사용자 테스트를 진행할 수 있는 서비스 수준이다.

## 2. 문제 정의

LLM 채팅 사용자는 어려운 개념, 복잡한 문제, 개발 또는 기획 과제를 다룰 때 긴 답변을 받는 경우가 많다. 이때 배경지식이 부족하거나 이해를 확인해야 하면 같은 채팅 안에서 꼬리질문이 반복된다.

꼬리질문이 누적되면 본래 목적과 직접 관련 없는 컨텍스트가 섞이고, 사용자는 원래 무엇을 하려 했는지 잊기 쉽다. LLM 역시 섞인 컨텍스트를 바탕으로 응답하게 되어 대화의 집중도와 문제 해결 효율이 낮아질 수 있다.

## 3. 타깃 사용자

- ChatGPT와 같은 LLM 채팅 서비스를 심도 있게 사용하는 사용자
- 학습, 개발, 기획, 연구 등 깊이 있는 사고가 필요한 사용자
- LLM 답변에 대해 반복적인 꼬리질문을 해야 하는 사용자
- 대표 사용자군: 학생, 개발자, 기획자, 연구자

라이트 유저는 핵심 타깃에서 제외한다. 이 서비스는 복잡한 대화 흐름을 관리해야 하는 파워 유저를 우선 대상으로 한다.

## 4. 기존 대안과 한계

ChatGPT나 Codex의 fork 기능은 대화 흐름을 분기하는 데 도움이 된다. 그러나 사용자가 브랜치를 손수 관리해야 하며, 브랜치가 어떤 대화 지점에서 파생되었는지 한눈에 파악하기 어렵다.

따라서 단순한 fork 기능만으로는 대화 집중도 문제를 충분히 해결하기 어렵다. 본 프로젝트의 차별점은 브랜치 생성뿐 아니라 브랜치 관계와 분기 지점을 시각적으로 관리하게 하는 UI에 있다.

## 5. 제안 솔루션

사용자는 LLM 채팅 중 특정 메시지에서 브랜치를 생성할 수 있다. 브랜치 안에서는 기존 대화의 맥락을 유지하면서도 본래 목적과 분리된 꼬리질문을 이어갈 수 있다.

서비스는 브랜치가 어느 메시지 또는 대화 흐름에서 분기되었는지 그래프 형태로 보여준다. 사용자는 필요 없는 브랜치를 비활성화하거나 접어서 숨김 처리하여 시각적 혼잡도를 낮출 수 있다.

MVP에서는 LLM API 연동을 염두에 둔 구조를 유지하되, 1차 검증의 초점은 채팅에서 브랜치를 생성하고 이를 시각화하는 UI에 둔다.

## 6. 핵심 기능

- LLM 채팅: ChatGPT, Gemini, Claude와 유사한 대화형 채팅 경험을 제공한다.
- 메시지 단위 브랜치 생성: 사용자는 특정 메시지에서 새 브랜치를 만들 수 있다.
- 브랜치 분기 지점 표시: 브랜치가 어느 메시지에서 생성되었는지 확인할 수 있다.
- 그래프 기반 시각화: 대화와 브랜치의 관계를 그래프 형태로 표시한다.
- 브랜치 비활성화: 필요 없는 브랜치를 삭제하지 않고 회색 표시, 접기, 숨김 등 낮은 강조 상태로 전환할 수 있다.
- 브랜치 클릭 이동: 사용자는 그래프에서 브랜치를 클릭하여 해당 브랜치 대화로 이동할 수 있다.
- 그래프 확대 및 축소: 브랜치가 많아지는 상황에 대비해 그래프를 확대하거나 축소할 수 있다.
- 브랜치 요약 미리보기: 사용자가 브랜치에 마우스를 올리면 LLM이 자동 생성한 해당 브랜치 대화 요약을 작은 미리보기로 볼 수 있다.

## 7. 사용자 흐름

1. 사용자가 LLM 채팅에서 본래 목적에 해당하는 질문을 입력한다.
2. LLM이 긴 답변을 제공한다.
3. 사용자가 답변 중 이해가 필요한 지점을 발견한다.
4. 사용자가 해당 메시지 또는 구간에서 브랜치를 생성한다.
5. 사용자는 브랜치 안에서 꼬리질문을 이어간다.
6. 사용자는 그래프 UI를 통해 현재 브랜치가 어디에서 분기되었는지 확인한다.
7. 사용자는 그래프에서 브랜치를 클릭하여 다른 브랜치 대화로 이동한다.
8. 사용자는 필요 없는 브랜치를 비활성화하거나 접어서 숨김 처리하고 본래 대화 흐름으로 돌아간다.

## 8. MVP 범위

- 기본 LLM 채팅 인터페이스
- 특정 메시지 기준 브랜치 생성
- 브랜치 분기 지점 기록 및 표시
- 그래프 기반 브랜치 시각화
- 브랜치 비활성화 및 회색 상태 표시
- 브랜치 클릭 이동
- 그래프 확대 및 축소
- 브랜치 마우스 오버 시 LLM 자동 요약 미리보기
- 비활성 브랜치 접기 및 숨김 처리
- LLM API 연동을 고려한 채팅 구조 설계

## 9. 범위 제외 항목

- 복수 LLM 제공자 전체 연동
- ChatGPT 확장 프로그램 구현
- 실제 서비스 수준의 LLM API 운영 안정화
- 협업 기능
- 브랜치 자동 추천
- 모바일 최적화
- 고급 검색 및 필터링
- 브랜치 병합 기능

위 항목은 MVP 이후 검토 대상으로 둔다.

## 10. 개발 고려사항

- 대화 메시지와 브랜치 간 부모-자식 관계를 안정적으로 저장해야 한다.
- 그래프 UI는 브랜치가 많아졌을 때도 사용자가 현재 위치와 원래 목적을 쉽게 파악할 수 있어야 한다.
- 비활성화는 삭제가 아니므로, 접힌 브랜치의 복구 가능 여부와 표시 방식을 별도로 정의해야 한다.
- MVP는 LLM API 연동을 염두에 둔 구조로 설계하되, 초기 검증은 UI와 브랜치 관리 기능에 우선순위를 둔다.
- 초기 버전에서는 하나의 LLM 제공자 또는 Mock LLM으로 시작할 수 있다.
- 그래프 UI는 디자이너에게 복수 시안을 요청한 뒤 사용성 기준으로 선택하는 것이 적절하다.
- 브랜치 요약 미리보기는 LLM 자동 요약으로 제공한다.
- 실제 사용자 모집, 사용자 테스트, 만족도 조사가 가능하도록 온보딩, 안정성, 기본 사용 흐름의 완성도를 확보해야 한다.

## 11. 리스크

- 브랜치가 많아질수록 그래프 UI가 복잡해져 오히려 집중도를 낮출 수 있다.
- 사용자가 브랜치를 생성해야 하는 시점을 이해하지 못하면 기능 사용률이 낮을 수 있다.
- 단순 fork 기능과의 차별점이 UI에서 충분히 드러나지 않으면 제품 가치가 약해질 수 있다.
- 이용 시간 증가는 긍정 지표일 수도 있지만, 사용자가 더 오래 헤매는 신호일 수도 있으므로 보조 지표가 필요하다.
- 마우스 오버 요약이 부정확하면 사용자가 브랜치 내용을 잘못 이해할 수 있다.
- 독립 웹 서비스로 구현하면 기존 ChatGPT 사용자 흐름에 바로 붙는 확장 프로그램 방식보다 초기 사용 진입 장벽이 높을 수 있다.
- 데모 수준을 넘어 실제 사용자를 모으는 경우, 인증, 데이터 저장, 오류 처리, 비용 관리 등 운영성 요구가 커질 수 있다.

## 12. 성공 기준

- 사용자 만족도 조사에서 브랜치 기능과 시각화 UI의 유용성이 확인된다.
- 브랜치 시각화 UI 도입 후 서비스 이용 트래픽이 증가한다.
- 브랜치 시각화 UI 도입 후 서비스 이용 시간이 증가한다.
- 사용자가 본래 대화 목적을 잃지 않는다고 응답하는 비율이 높아진다.
- 실제 사용자를 모집하여 사용자 테스트를 진행한다.
- 사용자 테스트 결과와 만족도 조사 결과를 확보한다.

## 13. 미확정 요구사항

- 그래프 UI의 구체적 레이아웃은 확정되지 않았다.
- 브랜치 생성 버튼의 위치와 인터랙션은 확정되지 않았다.
- 접기 또는 숨김 처리된 브랜치의 복구 방식은 확정되지 않았다.
- 초기 버전에서 실제 LLM API를 어느 수준까지 연동할지 확정되지 않았다.
- 성공 기준의 정량 목표값은 확정되지 않았다.
- 실제 사용자 모집 방식과 사용자 테스트 설계는 확정되지 않았다.

## 14. PRD 검토 결과

현재 PRD는 문제, 타깃 사용자, 현재 불편, 제안 솔루션, 핵심 기능, MVP 범위, 성공 기준, 초기 서비스 형태, 그래프 UI의 필수 조작, 브랜치 요약 방식, 사용자 검증 목표를 포함한다. 다만 그래프 레이아웃, LLM API 연동 수준, 비활성 브랜치 복구 방식, 사용자 모집 방식, 정량 목표가 아직 모호하므로 후속 인터뷰에서 보완해야 한다.
